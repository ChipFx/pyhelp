## Wishlist:
 - rst-doc links formed as `Import Menu <rst-doc://menu_bar/file_menu/import>`_. should be clickable and would ideally follow the treeview on the left into the main chapter code named 'menu_bar' and open the rst element 'import' under 'file_menu', but then that would imply a many-tiered structure, which is currently not supported by the rst structure/header information nor the yaml config. Until that multi level design is thought out, maybe it should just go as deep as it can in the treeview, relying on a future multi-level support. In this case ending up in file_menu from the chapter group menu_bar. At the moment it seems `<rst-doc://menu_bar/file_menu>`_ doesn't even parse if there is an RST file calling itself (short name) "file_menu" with as chapter short name "menu_bar", it just shows the exact string, rather than a parsed link-lookalike that isn't clickable. (`<rst-doc://menu_bar>`_ does look like it's clickable with the chapter menu_bar defined, but doesn't become clickable.)
 - An main-line supported method to create multi-tiered systems to link to. I'm already using the ==== and ---- underline for sub-chapter-title and paragraph in an rst, but that's limited "tiering" and not even officially supported, and I wouldn't know how to rst-doc-link to the ---- sub sections. Id's be good if you can link to more places than just chapter / sub-chapter. I would throw out ideas here, but it has to be compatible as much as possible with official RST based doc tools to generate PDF, HTML and searchable help indexes.
